# Vulkan Rendering Architecture (Current)

This project uses modern Vulkan with dynamic rendering and C++ RAII wrappers (vulkan-hpp vk::raii). 
The design avoids legacy render pass/framebuffer objects and focuses on clear ownership and testable components.

- Dynamic Rendering (VK_KHR_dynamic_rendering / Vulkan 1.3)
- RAII for all Vulkan resources (vk::raii::Device, Buffer, ImageView, etc.)
- Clear separation between “assets” (Shader, Mesh data) and “GPU objects” (pipelines, buffers)
- ResourceManager owns resources by value; others hold references

## Core Modules and Ownership

- VulkanDevice (engine/vulkan)
  - Owns instance, physical/logical device, queues, features (incl. dynamic rendering)
- VulkanSurface (engine/vulkan)
  - Wraps platform surface (Win32)
- VulkanSwapchain (engine/vulkan)
  - Owns swapchain, format/extent, and per-image vk::raii::ImageView
- VulkanCommandContext (engine/vulkan)
  - Per-frame command buffers, fences, semaphores (frames-in-flight)
- VulkanRenderer (engine/vulkan)
  - Creates and owns pipeline layout and graphics pipeline(s)
  - Orchestrates frame flow, dynamic rendering begin/end, submission, presentation
- GpuBuffer (engine/vulkan)
  - RAII buffer + memory allocation, upload helpers
- Vertex (graphics)
  - Uses glm types; provides binding/attribute descriptions for pipeline
- Mesh (graphics)
  - GPU-side mesh (vertex_count + GpuBuffer for vertices). Test factories available
- Shader (graphics)
  - Pure data: name + std::vector<uint32_t> SPIR-V + vk::ShaderStageFlagBits (no filesystem)
- ResourceManager (engine)
  - Owns Shaders and Meshes in unordered_maps keyed by name
  - API: load(...)/get<T>(...) (templated get; overloaded or templated load)
  - Loads SPIR-V via File helper; constructs Shader; creates Mesh from vertices
  - Returns T& (manager retains ownership; callers hold non-owning references)

## Frame Flow (High Level)

begin_frame():
- Wait for current frame fence
- Acquire next swapchain image → image index
- Reset fence, reset/re-begin command buffer

begin_rendering():
- Transition the acquired image to COLOR_ATTACHMENT_OPTIMAL (if not handled by implicit subpass; with dynamic rendering you add an explicit barrier if needed)
- Build VkRenderingInfo with the swapchain image view
- vkCmdBeginRendering

draw:
- Bind pipeline
- Bind vertex buffers
- vkCmdDraw (or vkCmdDrawIndexed)

end_rendering():
- vkCmdEndRendering

end_frame():
- Transition image to PRESENT_SRC_KHR (explicit barrier)
- End command buffer, submit with semaphores
- Present on the queue
- Advance frame index

## Graphics Pipeline (Dynamic Rendering)

- Pipeline layout: currently minimal/empty; grow with push constants/descriptors later
- Shader stages: vertex + fragment created from temporary vk::raii::ShaderModule
  - Modules are created during pipeline build from Shader.spirv() and destroyed immediately after
- Vertex input: from Vertex::get_binding_description() and get_attribute_descriptions()
- Input assembly: triangle list
- Viewport/scissor: dynamic states recommended
- Rasterization: fill, cull back, CCW
- Color blend: write to RGBA, blending off (enable later if needed)
- Multisample: 1x (no MSAA)
- Depth: none for Pong
- Dynamic rendering chain:
  - VkPipelineRenderingCreateInfo with colorAttachmentCount = 1 and swapchain format
  - Attach via pNext to VkGraphicsPipelineCreateInfo

## Dynamic Rendering Structures

Pipeline creation (pseudocode):
- VkPipelineRenderingCreateInfo renderingInfo{ colorAttachmentCount=1, pColorAttachmentFormats=&swapchainFormat }

Begin rendering (per frame/image):
- VkRenderingAttachmentInfo colorAttachment{ imageView=currentImageView, layout=COLOR_ATTACHMENT_OPTIMAL, loadOp=CLEAR, storeOp=STORE, clearValue=clear_color }
- VkRenderingInfo info{ renderArea.extent=swapchainExtent, layerCount=1, colorAttachmentCount=1, pColorAttachments=&colorAttachment }
- vkCmdBeginRendering(cmd, &info)

## Synchronization

- Frames-in-flight: typically 2
- Per-frame:
  - Fence: wait/reset per frame
  - Acquire semaphore (imageAvailable)
  - Render-finished semaphore (renderFinished)
- Submit: waits on imageAvailable, signals renderFinished
- Present: waits on renderFinished

## Assets and Resource Management

- Shader (data-only):
  - name: string
  - stage: vk::ShaderStageFlagBits
  - spirv: vector<uint32_t>
  - No filesystem or Vulkan dependency
- ResourceManager:
  - Loads SPIR-V using File helper (validation/utilities live in engine/file.*)
  - Constructs Shader and caches by name
  - Creates Mesh from vertex data (host-visible/coherent for now)
  - API shapes:
    - load(name, path, stage) -> Shader&
    - load(name, span<Vertex>) -> Mesh&
    - get<T>(name) -> T&; contains<T>(name) -> bool
  - Manager outlives users; consumers store references

## Mesh Handling
- Current Mesh class is GPU-side (vertex buffer + count)
- Memory properties (current):
  - HostVisible | HostCoherent for simplicity and quick uploads
- Future optimization:
  - DeviceLocal vertex buffer with staging buffer (TransferSrc → TransferDst copy)
- Roadmap:
  - Optional index buffer
  - MeshData (CPU-side) → Mesh (GPU-side) upload path
